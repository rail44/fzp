# REQUIREMENTS.md

`hunch` の機能要件。**MVP は Claude Code による実装時に具体化**する。このドキュメントでは方針を述べ、具体的なフラグ名や挙動の詳細は敢えて書かない。細部は実装しながら決めるべきものが多い。

## MVP(最初のリリースで実装)

### R-1: Unix パイプフィルタとしての基本動作

- stdin から入力を読む(デフォルト)
- stdout に結果を書く(デフォルト)
- stderr に進捗とエラーを書く
- SIGPIPE を正しく処理する(下流が終了したら速やかに停止)
- 行指向の逐次処理(全件メモリに読み込んでから処理しない)
- 終了コードの規約:
  - 全件成功: 0
  - 個別失敗あり・処理完了: 0(失敗は別ストリームに)
  - 致命的エラー(引数不正、API キー無しなど): 非ゼロ

### R-2: 入出力フォーマット

- 入出力ともに **JSONL** を基本とする
- 入力の各行には「識別子」と「推論対象のテキスト」を含む
- 出力の各行には「識別子」と「推論結果」を含む
- 詳細なスキーマ(フィールド名、必須/任意、エスケープなど)は実装時に決定する
- 方針: **最小主義**。余計なメタデータは入れない。必要なら後から拡張可能な形にする

### R-3: 並列実行

- 複数リクエストを並行して LLM API に投げる
- 並列度は CLI フラグで制御可能
- デフォルト並列度は実装時に決定(控えめな数で開始する方針)
- レート制限を考慮し、429/5xx は指数バックオフで自動リトライ

### R-4: OpenAI 互換エンドポイント呼び出し

- OpenAI Chat Completions API 互換のエンドポイントを叩く
- エンドポイント URL、モデル名、API キーの読み方(環境変数名)は CLI フラグか環境変数で指定可能
- デフォルトは Gemini 3.1 Flash-Lite の OpenAI 互換エンドポイントを想定
- 認証は Bearer トークン(OpenAI 互換の標準)

### R-5: タスクプリセット

よく使うタスクのプロンプトテンプレートをビルトインで持つ。MVP で対応する範囲:

- **分類(classify)** — ラベル集合を与え、1 件につき 1 ラベルを返させる
- **抽出(extract)** — 構造化 JSON を返させる
- **要約(summarize)** — 短い自然文で要約させる
- **翻訳(translate)** — 指定言語に翻訳させる
- **カスタム(custom)** — ユーザー指定プロンプトで処理する

各プリセットのプロンプト詳細は実装時に決定する。**プリセットの追加・改変は将来も続くので、拡張しやすい構造にする**(例: プリセット定義ファイルを外出しする、など)。

### R-6: 失敗の分離

- 成功結果は stdout(または `--output` 指定ファイル)に書く
- 失敗したアイテムは別ストリーム(`--failures` 指定ファイル)に記録する
- 失敗は全体処理を止めない
- 詳細な失敗情報フォーマットは実装時に決定

### R-7: 進捗表示

- stderr に進捗情報を出す(stdout を汚さない)
- 処理件数、成功数、失敗数を表示
- 詳細な表示形式は実装時に決定(シンプルな 1 行更新で十分)

### R-8: エラーハンドリング

- 致命的エラーは stderr に表示して非ゼロで終了
- 個別アイテムの失敗は失敗ストリームに記録して処理継続
- パニックやハングで止まらないこと

## 将来機能(運用で必要性を見てから実装)

優先度順。上から順に検討する。

### F-1: ショートカット入力モード

stdin JSONL 以外にも、以下のような入力方法があると便利かもしれない:

- CLI 引数で直接テキストを渡す(10 件未満の書き捨て用)
- ファイルグロブから自動生成(`--from-files 'src/**/*.ts'` のような形)

ただし **フラグが増えすぎる危険** を強く意識すること。パイプから JSONL を受けるのが基本形で、ショートカットは明らかに頻用されるものだけ追加する。

**採用基準**: 実運用で JSONL 手動整形が面倒だと感じる頻度が高くなったら。

### F-2: レジューム対応

途中で止まったジョブを `--resume` で続きから再開できる。

- 処理済み ID の記録方法(SQLite 埋め込み、JSONL マーカー、外部 DB など)は実装時に検討
- **採用基準**: 1 ジョブが 10 分を超えるのが日常になったら

### F-3: コスト見積もり

`--dry-run` で実行前に推定コストとトークン数を表示。

- モデルごとの料金テーブルが必要
- **採用基準**: 「このジョブ走らせて大丈夫?」と迷う頻度が高くなったら

### F-4: サンプリング

`--sample N` で最初の N 件だけ実行。プロンプト妥当性の事前確認用。

- **採用基準**: プロンプト調整を繰り返すフェーズに入ったら

### F-5: キャッシュ

入力ハッシュで結果をキャッシュし、同一入力は再リクエストしない。

- ストレージの場所、TTL、モデル+プロンプト変更時の扱いは実装時検討
- **採用基準**: 同じ入力セットで何度も再実行するようになったら

### F-6: Batch API 対応

`--batch` で OpenAI 互換 Batch API に投げる。50% 割引と rate limit 回避の両得。

- **採用基準**: 1 ジョブが 1000 件を超えるのが日常になったら

### F-7: サマリーモード

全件の結果に加えて「全体のサマリー」を 1 つ追加生成する。

- 親 Claude Code がサマリーだけ読めば済むフロー用
- **採用基準**: 大量結果を Claude Code が処理する頻度が高くなったら

### F-8: 評価モード

複数モデル・プロンプトを横並びで評価する(正解ラベル付きデータセットに対する一致率、コスト、レイテンシ)。

- **採用基準**: モデル比較やプロンプト調整を頻繁にやるようになったら

### F-9: プロバイダー抽象化

OpenAI 互換に限らない(Anthropic ネイティブ、Vertex AI など)プロバイダー切り替え。

- トレイトで抽象化し、条件付きコンパイルでビルド時選択できる形が素直
- **採用基準**: 複数プロバイダーを日常的に切り替えるようになったら

### F-10: 構造化ログ / 可観測性

`tracing` + OpenTelemetry でメトリクス送信。

- 導入コストは低いが、運用で使わないなら無駄
- **採用基準**: 可観測性が本気で必要になったら

## スコープ外(やらない)

以下は意図的に対象外。要望が来ても実装しない。

### OOS-1: コード変更機能

整合性・責任・可観測性の問題。親 Claude Code の `edit` ツールで行う。

### OOS-2: マルチターン対話

単ターンのファンアウト専用。対話が必要なら親 Claude Code で。

### OOS-3: エージェント的ループ

ツール呼び出しや外部 API 連携を伴うループは扱わない。MCP サーバーなど別物として実装すべき。

### OOS-4: Web UI / ダッシュボード

CLI に徹する。可視化は出力 JSONL を別ツールで処理すればよい。

### OOS-5: 独自のモデルホスティング

Ollama 的な自前サーバーは立てない。既存の OpenAI 互換エンドポイントを叩くことに徹する。

## 設計判断の指針(実装時の参照)

実装で迷った時の優先順位:

1. **シンプルさ > 機能の豊富さ** — フラグの追加は慎重に。既存の組み合わせで実現できないか先に考える
2. **Unix 作法 > 独自規約** — stdin/stdout、終了コード、SIGPIPE、パイプ連結
3. **ストリーミング > バッチ** — 逐次処理を基本に設計する
4. **明示 > 暗黙** — デフォルトよりもフラグを要求する判断もアリ(特に誤用の代償が大きい場合)
5. **失敗しやすい箇所は丁寧に** — HTTP エラー、JSON パース、I/O は詳細にハンドリング
6. **Claude Code に改修させやすく** — コード構造は平易に、抽象化は必要になってから

## Bun 版を作らない方針の確認

議論の経緯で「Bun で始めてから Rust に移行」という案もあったが、採用しない。理由:

- 仕様の核(パイプフィルタ、JSONL、並列)は既に固まっている
- 実装は Claude Code が書くので、人間の言語習熟度は律速にならない
- 移行コスト(書き直し)を避けられる
- Rust の弱点(反復サイクルの遅さ)は Claude Code 運用で緩和される

仕様が動的に変わる部分(プロンプト、プリセット詳細、フラグ名)は、Rust 実装の中でも**変更しやすい箇所に閉じ込める**設計にすることで対応する。
