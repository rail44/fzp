# hunch Skill

大量の同質タスクを低コスト LLM で並列処理する Unix パイプフィルタ。

`hunch` は「直感、軽い判断」を意味する英単語で、軽量モデルが出す「大量の小さな直感」をパイプで流し込んで並列に得る道具。

## 位置づけ

- **親 Claude Code** — パイプラインの設計、意思決定、コード変更
- **`hunch`** — 読み取り系の map 操作を軽量モデルに剥がす
- **jq/rg/awk/sed** — 前処理・後処理を担う

## 基本的な使い方

### パイプで JSONL を流す(基本形)

```
<データ源> | <前処理> | hunch <タスク指定> | <後処理>
```

具体例(**正確なフラグ名は実装後の `docs/CLI_REFERENCE.md` を参照**):

```bash
# 関数群の一行要約(イメージ)
rg --json 'fn |function ' src/ \
  | jq -c '...前処理でJSONL化...' \
  | hunch ...(要約タスク指定) \
  | jq -c '...後処理で集約...'
```

### JSONL ファイルを指定

```bash
hunch --input items.jsonl --output results.jsonl ...(タスク指定)
```

### 失敗を分離

失敗したアイテムは別ファイルに記録される(詳細は CLI_REFERENCE.md)。失敗分だけ取り出して再実行する運用ができる。

## タスクの種類

以下のプリセットを想定している(詳細は実装時決定):

- **分類** — ラベル集合を与えて 1 ラベル返させる
- **抽出** — JSON スキーマ的に構造化データを返させる
- **要約** — 短い自然文の要約
- **翻訳** — 対象言語への翻訳
- **カスタム** — ユーザー指定プロンプト

## 判断チェックリスト

このスキルを呼ぶ前に、以下を確認する:

- [ ] タスクが 10 件以上ある
- [ ] 各タスクが独立している
- [ ] 1 件あたり 5 秒未満で済む軽い判断
- [ ] 結果が構造化可能(ラベル/JSON/短文)
- [ ] コード変更ではない(読み取り系のみ)
- [ ] 親 Claude Code の文脈に全結果を載せる必要がない

6 項目すべて YES なら剥がすべき。いずれかが NO なら親 Claude Code で直接処理する。

## パイプライン設計のコツ

### 前処理(データを JSONL にする)

- `rg --json` — ripgrep の JSON 出力を jq で整形
- `find ... | xargs` で複数ファイルを処理
- `git log --format=...` + awk で履歴データを作る
- `cat file.json | jq -c '.items[]'` で既存データを 1 行ずつ展開

### 後処理(結果を集約する)

- `jq -s 'group_by(.result) | ...'` — 結果ラベルで集計
- `jq -c 'select(.result.risk == "high")'` — 重要なもののみ抽出
- サマリーだけ親 Claude Code に戻す(全件載せない)

### パイプ中のデバッグ

- `| head -3` を挟んで途中結果を見る
- `| tee /tmp/debug.jsonl` で中間結果を保存しつつ流す

## 前提条件

- 実行バイナリ `hunch` が PATH に通っているか、プロジェクト内にビルド済み
- API キーが環境変数に設定されている(変数名は実装後の CLI_REFERENCE.md 参照)
- 入力が適切な JSONL 形式である

## 失敗時の扱い

- 個別失敗は失敗ストリームに記録され、処理全体は継続
- リトライは自動(429/5xx は指数バックオフ)
- 致命的エラー(API キー無しなど)は即終了
