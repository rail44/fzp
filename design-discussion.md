# 設計議論ログ

`hunch` プロジェクトの設計に至るまでの議論を、**判断の「なぜ」を中心に**記録したもの。設計判断で迷った時はこのドキュメントを参照し、過去の議論と一貫性のある判断をすること。

新しい Claude Code セッションでは、**このドキュメントを最初に読ませる**こと。`DESIGN.md` や `REQUIREMENTS.md` は結論だけを書いているので、「なぜそうしなかったか」が分からないと拡張提案がスコープアウトしがち。

---

## 1. 出発点: 問題の定義

### 問い

分量は多いが判断の難易度は高くないタスクを、小さく安価で高速なモデルで並列実行するにはどうすべきか。

### 背景

- Anthropic の Message Batches API は 50% 割引、最大 100,000 リクエスト/バッチ、多くは 1 時間以内に完了
- Haiku 4.5 は最小・最速・最安($1/M input, $5/M output)
- プロンプトキャッシング併用で追加 90% 削減可能

### 初期方針

- 時間にシビアでなければ Batches API + Haiku 4.5
- リアルタイムが必要なら同期 API の並列実行
- 共通プロンプト部分はキャッシング

---

## 2. 認証モデルの制約

### 発見

- Claude Code が `ANTHROPIC_API_KEY` 環境変数で動いている場合は SDK が自動で読むので引き継げる
- Pro/Max/Team サブスクリプションで `/login` している場合は OAuth で、推論専用にスコープされており Batches API に使えない
- `claude setup-token` で生成する `CLAUDE_CODE_OAUTH_TOKEN` も推論専用

### 結論

Batches API を使うには Anthropic Console で別途 API アカウントを作り API キーを発行する必要がある。サブスクリプション枠内では Batches は使えない。

---

## 3. OpenRouter とプロバイダー選定

### 問い

Pro プラン枠で Batches が使えないなら、OpenRouter 等から低コスト・高速なモデルを選定する方が合理的か。

### OpenRouter の実態

- プロバイダー価格をマークアップなしで通す
- クレジット購入時に 5.5%(最低 $0.80)の手数料
- Batch API の 50% 割引は基本的に効かない
- モデルのベンチマーク・評価用途では優秀
- 本番の単一モデル運用では直接契約の方が安い

### 結論

OpenRouter はモデル選定の「実験台」として使い、本番は直接契約 + Batch API。

## 4. Haiku の上位互換候補

### 2026 年 4 月時点の候補比較

| モデル | 入力 $/M | 出力 $/M | 備考 |
|---|---|---|---|
| Claude Haiku 4.5 | $1.00 | $5.00 | 基準 |
| GPT-5.4 mini | $0.75 | $4.50 | 2026/3 リリース |
| GPT-5.4 nano | $0.20 | $1.25 | 軽量タスク向け |
| Gemini 3 Flash | $0.50 | $3.00 | 高スループット |
| Gemini 2.5 Flash-Lite | $0.10 | $0.30 | 分類・抽出最安枠 |
| DeepSeek V3.2 | $0.28 | $0.42 | 価格破壊枠 |
| Mistral Small 3.2 | $0.06 | $0.18 | オープンウェイト |

### 用途別の勝者

- コーディング・推論含む: GPT-5.4 mini
- 純粋な分類・抽出: Gemini Flash-Lite(Haiku の 1/10 コスト)
- エージェント・ツール呼び出し堅牢性: Haiku が残る
- 日本語タスク: Gemini 系が比較的強い

### 結論

用途を絞れば Haiku の上位互換は複数存在する。「大量・簡単判断」では Gemini Flash-Lite / GPT-5.4 nano が本命。

## 6. Claude Code 連携方式の検討

### 重要な誤解の訂正

Claude Code の「サブエージェント」は Claude インスタンスを起動する機構であり、外部モデル(Gemini 等)を呼ぶためのものではない。サブエージェント機能は今回のユースケースには使わない。

### 候補パターン

1. Bash ツールで直接スクリプト実行 — 最小構成、試作向き
2. Skill 化 — 再利用可能
3. MCP サーバー化 — 第一級のツール、セットアップコスト大

### 結論

再利用性とシンプルさのバランスで **Skill 方式**。MCP は複数プロジェクトで使うようになったら検討。

## 7. Ollama の Claude Code subagent 対応調査

### 実装の仕組み

- Ollama は `/v1/messages` で Anthropic Messages API 互換エンドポイントを提供
- Claude Code は `ANTHROPIC_BASE_URL=http://localhost:11434` を向ければ Ollama を叩く
- ツール呼び出し変換層が Anthropic ⇔ Ollama 内部形式をマッピング
- **サブエージェント起動自体は Claude Code のクライアント側で完結**
- Ollama の貢献は「モデルを提供」と「ツール呼び出しのパススルー品質向上」

### Web 検索の実装

`WebSearchAnthropicWriter` がサーバー側で多ターンのツールループを閉じる。モデルが検索ツール呼び出しを出すと Ollama が検索実行、結果を再注入、最終回答を生成する。

## 8. 互換レイヤー方式 vs Skill 方式

### 問い

OpenAI 互換 LLM で並列タスクをさせる時、Ollama 的な互換レイヤーを作るか、Skill として提供するか。

### 互換レイヤー方式の問題点

解く問題が違う:

- 互換レイヤー = Claude Code のオーケストレーター自体を別モデルで置き換える
- 今回のユースケース = Claude をオーケストレーターに残したまま、大量の単純タスクを安いモデルに投げる

互換レイヤーを無理に使うと:

1. Claude Code のシステムプロンプト 18K+ トークンが毎回乗る
2. 並列度が Claude Code クライアント依存
3. Batch API の 50% 割引が使えない
4. 軽量モデルが `Task` ツール呼び出しプロトコルを踏めるか未知数
5. 実装が数週間〜数ヶ月(Messages API 完全再現が必要)

### 結論

**Skill 方式が圧倒的に適合**。オーバーヘッド小、並列度自由、Batch 対応可能、実装数時間。互換レイヤーは大砲で蚊を撃つ構造。

## 9. 実行環境の初期検討

### 制約

Python を避けたい(チームの嗜好)。

### 候補評価

- Bash + curl: 100 行以内なら OK、複雑になると辛い
- Deno: 良い選択肢だが Bun に劣後
- **Bun**: 起動速度、npm 互換、TS ネイティブ — Bun が第一候補に
- Node + TS(直接実行): ゼロ依存原理主義なら
- Go: スクリプト感覚で書けない
- Rust: 当初の評価では過剰投資

### 当初の結論

Bun を選定。Claude Code との相性、npm エコシステム、起動速度で最適と判断。

## 10. Rust バイナリ化の初期議論

### 問い

ネイティブ言語でバイナリ化すれば便利機能を作り込む余地があるのでは?

### Rust で効くとされた機能

- `tower` での堅牢なリトライ・サーキットブレーカー
- SQLite 埋め込みでレジューム対応
- `trait Provider` による複数プロバイダー抽象化
- `tracing` + OpenTelemetry の可観測性
- ストリーミング処理(メモリ一定)
- `indicatif` での TUI 進捗
- 単一バイナリ配布
- `serde` でのスキーマ検証

### 当初の却下理由

- スコープを「読み取り系の map 操作」に絞ったため数万件規模は想定外
- 要件が動的
- Claude Code による改修のしやすさで TypeScript が勝る
- Bun で 200〜300 行で済む

### 当初の結論

段階制で進める。まず Bun、運用で要件固まってから Rust 検討。

## 11. 作業パターンの分類

### 軽量 LLM 並列ファンアウトの 3 条件

1. 同質のタスクが多数
2. 各タスクが独立
3. Claude Code の文脈に全結果を載せる必要がない

### 有用なパターン(A〜E)

- A: コードベース横断の分類・抽出
- B: ドキュメント大量変換
- C: 評価・レビューのファンアウト
- D: データ前処理・クリーニング
- E: 検証・照合

### 共通構造

すべて **map 操作**。Claude Code は reduce と全体判断に集中、このツールは map を剥がす。

## 12. 責任分界: コード変更は親が行う

### 問い

実際のコード変更は親 Claude Code のツール呼び出しか、分離環境の軽量 LLM か。

### 軽量 LLM にコード変更を任せない理由

1. **失敗コストの性質が違う** — 分類の失敗は統計的に許容、コード変更は離散的(1 ファイル壊れたらビルド全滅)
2. **ローカルに閉じない変更の整合性** — シグネチャ変更は呼び出し側にも波及、独立モデルは他タスクの変更を知らない
3. **責任と可観測性の分界** — 親に集約されていないと、変更理由の把握・レビュー・ロールバックができない

### 軽量 LLM に向いているコード関連タスク

- 読み取り: 関数の責務抽出、呼び出し関係整理、既存スタイル棚卸し
- 提案(適用はしない): 変更案生成を JSON で返す、親が判断して適用
- 検証: 変更後のセルフレビュー、テスト失敗パターン分類

### 例外: 完全独立な機械的編集

以下は剥がしてよい:

- 1 ファイル内・1 箇所のみ・他と依存なし
- 結果を機械的に検証可能
- 失敗時のロールバックコストが低い(Git 前提)

例: i18n キーの一括翻訳、コメント翻訳、ログフォーマット統一。

### 分離環境(agentfs 等)について

ファイルシステムレベルの隔離は、整合性・責任・可観測性の問題を解決しない。副作用の隔離や並列実験には有効だが、軽量 LLM の並列編集問題には効かない。

## 13. 既存品調査

以下の既存ツール/Skill/プラグインを調査:

### 特に近い既存品

- **Claude Code built-in `/batch`** — Claude 自身の git worktree 並列、このツールとは目的が違う
- **`s2-streamstore/claude-batch-toolkit`**(`/batchy`) — Anthropic Batch API 専用、Skill+MCP+Status Line

「Claude Code の Skill として動作し、JSONL を受け取って OpenAI 互換エンドポイントにバルク並列リクエストを投げ、構造化結果を返す」というピンポイントの組み合わせは空白。

### 結論

自作する価値がある。ただし `claude-batch-toolkit` の構成(Skill + MCP + Status Line)は将来 Batch API 対応時の参考になる。

## 14. パイプフィルタ設計への転換

### 問い

並列タスクを実行する時、Claude Code の作業フローはどうなるか。スクリプトが主体か、ツールが解釈するか、パイプで使うか。

### 3 つのパターン

- **X**: スクリプトに全てを書かせる — Claude が前処理を書く
- **Y**: 宣言的に引数として与える — ツールがデータソースを解釈
- **Z**: Unix パイプ思想で stdin/stdout — フィルタプログラム

### 決定的な観点: Claude Code の得手不得手

- 得意: Unix パイプラインの組み立て(rg | jq | xargs のパターン)
- 苦手: 複雑な CLI フラグの組み合わせ暗記、ツール固有スキーマの暗記

### 結論

**パターン Z(Unix パイプフィルタ)** を主軸にする。Claude Code の得意技(シェルパイプ構築)を活かし、苦手(固有フラグ暗記)を避ける設計。

### 追加の洞察

「常に stdin」を強制すると書き捨ての小規模タスクで冗長になるため、以下のショートカットは将来検討(MVP には入れない):

- `--from-files` でファイルグロブから自動生成
- 引数で直接テキストを渡す

## 15. Rust 再評価

### パイプフィルタ化で変わった要素

1. **ストリーミング処理が本質になる** — SIGPIPE 対応、backpressure、バッファ管理を正しくやる必要
2. **起動時間が効く場面が増える** — パイプの中で叩かれる可能性
3. **パイプフィルタは機能的にシンプル** — Rust の苦手な領域(動的 UI、対話)は要件に入らない
4. **バイナリ配布の摩擦が Bun より低い** — 単一バイナリ vs `bun install`

### Bun 依然有利な点

1. Claude Code による改修のしやすさ
2. ビルドサイクルの速さ
3. コード量の少なさ
4. 外部依存の少なさ

### 当初の再評価結論

Bun で始めて、要件が固まったら Rust 移行を検討。パイプフィルタ化により移行の妥当性は確実に上がっているが、まだ Bun が有利。

---

## 16. 最終決定: 最初から Rust

### 再々評価の理由

Bun → Rust の移行を挟む案は、以下の理由で最終的に採用せず:

- **仕様の核は既に固まっている** — パイプフィルタ、JSONL、並列制御、OpenAI 互換エンドポイント、タスクプリセット
- **実装は Claude Code が書く** — 人間の Rust 習熟度は律速にならない
- **移行コストが無駄** — 仕様が明確でも書き直しは非自明な手戻り
- **Rust の弱点は Claude Code 運用で緩和** — 反復サイクルの遅さはコード生成の速さで相殺

### 動的に変わる要件への対処

以下は実装時に Claude Code が決定し、必要に応じて変更しやすい形で設計:

- CLI フラグ名
- JSONL の詳細スキーマ
- タスクプリセットのプロンプト内容
- デフォルト値(並列度、モデル ID、失敗ファイル名など)

これらを事前に固定しない。方針だけ `INTERFACE.md` に書き、具体は実装中に最適化する。

### 結論

**最初から Rust で実装**。

## 17. ツール名の選定

### 命名の候補変遷

最初に検討した候補:

- `llmap`、`fanout`、`offload`、`chore`、`sift`、`distill`、`glean`、`scout`

却下された方向性:

- `llmap`、`aimap` — 「LLM」「AI」は実装詳細の話で、ツールの目的からは外れる
- `distill` — 機械学習の knowledge distillation と衝突
- `fanout`、`offload`、`chore` — 「推論」「思考」のニュアンスがない

### 最終的な命名の軸

3 軸すべてを 1 語で含む:

- **推論・思考**(LLM の機能ではなく、ツールが解決する目的)
- **並列展開**(パイプから多数へ)
- **軽量性**(大仰な推論でなく、軽い判断)

### 採用: `hunch`

「直感、軽い推論」を意味する。3 軸をすべて含む:

- **推論・思考**: hunch は判断の一種
- **並列展開**: 「多数の hunch を出す」と自然に読める
- **軽量性**: deep reasoning ではなく、直感レベル

他の候補との比較:

- `sense` — 3 軸は満たすが一般語すぎて検索性低い
- `deem` — 軸 B(並列)が名前に出ない
- `chime` — chime in の連想で並列だが、意味がやや弱い

`hunch` の強み:

- 5 文字と短い(Unix ツール感)
- 「軽量モデルが出す大量の直感」という本質を最も正確に表現
- 既存 CLI ツールと未衝突
- 日本語話者にも「ハンチ」として通じる
- LLM・AI という実装語を含まないので、将来技術が変わっても名前が古びない

## 18. 進行戦略

### フェーズ 1: 設計文書の凝固(完了)

- このスレッドで議論した内容を `docs/` に落とす

### フェーズ 2: Claude Code セッションでの実装

- 実データで試しながらプロンプト調整
- 運用で必要になった機能を追加(`REQUIREMENTS.md` の F-1 〜 F-10)
- 仕様の変更は `INTERFACE.md` や別ファイルに記録

## 付録: 捨てた選択肢の記録

後から「あの選択肢は?」と迷わないよう、検討したが採用しなかったものを記録する。

- **Python + asyncio** — 言語嗜好で却下
- **Deno** — Bun に劣後(当初)
- **Bun(最終)** — Rust に移行する手間を避けるため、最初から Rust に
- **Go** — スクリプト感覚では書けない、Bun の軽快さがない
- **Anthropic 互換レイヤー(自作 Ollama 風)** — 解く問題が違う、オーバーヘッド大
- **MCP サーバー(初期から)** — 単一プロジェクトではオーバーキル
- **Claude Code のサブエージェントで外部モデル呼び出し** — サブエージェントは Claude インスタンス起動機構
- **全件を Claude Code に直接処理させる** — コンテキスト汚染とコスト非効率
- **`fanout`、`offload`、`chore`、`distill` 等のツール名** — 推論・思考のニュアンスに欠ける、または実装語に寄っている
