# INTERFACE.md

`hunch` の CLI インターフェース設計方針。**具体的なフラグ名や入出力フォーマットは Claude Code による実装時に決定する**。このドキュメントは「どういう形にすべきか」の方針を示すもの。

## 基本形

```
<データ源> | hunch [タスク指定] [オプション] > <出力ファイル>
```

パイプの中で動作することを最優先する。sed、awk、jq と同じ流儀。

## 入力方針

### 主: stdin からの JSONL

**これが基本形**。stdin から 1 行 1 JSON オブジェクトを逐次読む。

```bash
cat items.jsonl | hunch ...
```

親 Claude Code は jq や rg と組み合わせて自由に前処理する:

```bash
rg --json 'pattern' src/ | jq -c '{...}' | hunch ...
```

### 副: ファイル指定

明示的に書きたい場合、stdin の代わりにファイルからも読める。

```bash
hunch --input items.jsonl ...
```

挙動としては stdin と同じ(中身のセマンティクスに差異はない)。

### 将来検討: ショートカット入力

以下は F-1 として将来検討する候補。**MVP では実装しない**。

- CLI 引数に直接テキストを渡す(10 件未満向け)
- ファイルグロブから自動生成(`--from-files 'src/**/*.ts'`)

これらを MVP に入れないのは、フラグの種類が増えると Claude Code が選択に迷うため。まずはパイプ 1 本で動く状態を目指す。

## 出力方針

### stdout — 成功結果の JSONL

デフォルトで結果を stdout に書く。1 行 1 JSON オブジェクト。

```bash
cat items.jsonl | hunch ... > results.jsonl
```

### `--output` ファイル指定

明示的にファイル指定したい場合:

```bash
hunch ... --output results.jsonl
```

### stderr — 進捗とメッセージ

進捗、警告、エラーメッセージは stderr に書く。stdout は汚さない。

### 失敗ストリーム

個別アイテムの失敗は、stdout とは別のファイルに記録する。デフォルト値とファイル名の規約は実装時決定。フラグで場所指定可能にする。

## JSONL スキーマの方針

**具体的なフィールド名は実装時に決定するが、以下の方針に従う**:

### 入力

- **必須**: 識別子フィールドが必要(結果との突き合わせのため)
- **必須**: 推論対象のテキストを含むフィールド
- **任意**: プロンプトテンプレートで参照できる追加コンテキスト

フィールド名は**短く**、**一般的な名前**を選ぶ(独自の用語や略号を避ける)。既存の類似ツールの慣習に乗るか、明確に一般的な語を使う。

### 出力

- **必須**: 入力の識別子を含む(突き合わせ可能にする)
- **必須**: 推論結果
- **任意**: メタデータ(モデル名、トークン数、レイテンシなど)— デフォルトでは付けない

出力はパース容易であること。つまり:
- ネストを浅く
- フィールド名は固定
- `null` や欠損の扱いを明確にする

### 失敗ストリーム

- **必須**: 入力の識別子
- **必須**: エラー内容(分類可能な形式、例えばエラー種別と詳細メッセージ)
- **任意**: 再試行回数やレスポンスボディなど

## CLI フラグの方針

### フラグ名の規約

- 長い形式(`--foo-bar`)が基本
- 短縮形は頻用されるものだけに限定(`-h`、`-v` など標準的なもの)
- 否定形(`--no-xxx`)は必要最小限
- ブール値は原則「指定すれば有効」(デフォルト false、フラグで true)

### 必要最小限のフラグ群

MVP で必要になるフラグのカテゴリ:

- **タスク指定**(プリセット名、カスタムプロンプト、ラベル集合など)
- **モデル指定**(モデル名、エンドポイント URL、API キー環境変数名)
- **並列度**(同時リクエスト数)
- **入出力**(入力ファイル、出力ファイル、失敗ファイル)
- **ヘルプ**(`-h` / `--help`)

それ以外のフラグは MVP では持たない。増やすのは運用で必要性が見えてから。

### 具体的なフラグ名は実装時決定

フラグ名は Claude Code が実装する過程で、以下を考慮して決定する:

- Unix ツールの慣習(`--verbose`、`--quiet`、`--output` など)
- `clap` の推奨規約
- 既存の類似 CLI(`jq`、`llm`、`gh` など)との整合
- 短くて覚えやすいこと

## パイプラインでの振る舞い

### SIGPIPE 対応

下流プロセスが終了した時(`| head -5` など)、速やかに処理を止めて正常終了する。インフライトのリクエストはキャンセルするか、結果を捨てる。

### バッファリング

stdin の読み込み・stdout への書き込みともに、**逐次的に流す**。行単位のバッファリングが基本。全件処理してから出力する実装は避ける。

### エラー時の振る舞い

- stdin が壊れた JSONL を含む: 該当行をスキップして継続(stderr に警告)。致命的ではない
- API キーがない: 致命的エラーで即終了
- 全リクエストが失敗する: 致命的エラーではないが、stderr に警告を出す

## プロバイダーと認証

### デフォルト

OpenAI 互換の Chat Completions エンドポイントを叩く。デフォルトモデルは Gemini 3.1 Flash-Lite 想定。

### 認証

Bearer トークン。API キーは環境変数から読む。デフォルトの環境変数名は実装時決定(例: `HUNCH_API_KEY` のようなツール固有の名前か、`GEMINI_API_KEY` のようなプロバイダー固有の名前か、どちらが望ましいかは実装時に判断)。

### URL 構築

OpenAI 互換エンドポイントは `POST {base_url}/chat/completions` が基本。`base_url` のスラッシュの有無の扱いは実装時に丁寧に決める(よくある不具合源)。

## 将来の拡張余地

以下は MVP に含めないが、CLI 設計時に拡張余地を残しておく:

- サブコマンド形式への移行(`hunch run`、`hunch eval`、`hunch batch` のような形)— 評価モードや Batch API 対応時に検討
- 設定ファイルのサポート(`.hunchrc` や `hunch.toml`) — プロファイル定義が必要になったら
- プラグイン機構 — タスクプリセットを外部化したい要望が出たら

MVP ではフラットな単一コマンドで始める。サブコマンド化はそれ自体が仕様変更なので、必要になってから行う。

## 参考にする既存ツール

CLI 設計の参考として、以下のツールの挙動は知っておくとよい:

- **`jq`** — パイプフィルタの模範。stdin/stdout、行指向、エラー出力の分離
- **`rg`(ripgrep)** — 高速な Rust 製 CLI の模範。`clap` の使い方
- **`simonw/llm`** — LLM CLI の先行例。プロバイダー抽象化、プロンプトテンプレート
- **`gh`** — サブコマンド設計の模範(将来サブコマンド化する場合)
- **`xargs`** — 並列実行の古典。`-P` の挙動

これらの良い点を取り入れ、独自に作らない。

## 実装者(Claude Code)への指示

このドキュメントを読んだ上で、以下を決定・実装してほしい:

1. **正確なフラグ名**(このドキュメントの方針に従いつつ、Rust の `clap` と相性の良い名前)
2. **JSONL の具体的なフィールド名**(短く、わかりやすく、類似ツールの慣習に沿う)
3. **タスクプリセットのプロンプト内容**(軽量モデルで動く簡潔なもの)
4. **デフォルト値**(並列度、失敗ファイル名、モデル ID など)
5. **エラー種別の分類**(致命的 / 個別失敗 / 警告)

決定した内容は、この `INTERFACE.md` 自体を更新するか、`docs/CLI_REFERENCE.md` のような別ファイルに記録すること。このファイルは「方針」を保つので、実装の結果として決まった仕様は別に残すのが望ましい。
