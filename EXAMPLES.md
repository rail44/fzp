# EXAMPLES.md

`hunch` を使うべき代表的なパターン。**コマンドの細部(フラグ名、引数形式)は実装時に決定する**ため、このドキュメントの例は「こういう使い方をする道具である」ことを示す概念的なもの。実際の構文は実装後に別ドキュメント(CLI_REFERENCE.md など)で記述する。

## 適合条件

`hunch` を使うべきタスクは、以下 3 条件を満たすもの:

1. **同質のタスクが多数ある**(1 件のロジックが全件で同じ)
2. **各タスクが独立**(並列実行できる、順序依存なし)
3. **Claude Code の文脈に全結果を載せる必要がない**(サマリーだけでよい)

この条件で分類したパターンを示す。

## パターン A: コードベース横断の分類・抽出

大規模リポジトリで親 Claude Code が全体像を把握する前段として使う。

### 例 A-1: 全関数に一行要約を付ける

```
rg で関数定義を抽出
  ↓
jq で JSONL に整形(識別子=ファイル:行、本文=関数)
  ↓
hunch で要約タスク(軽量モデル、並列)
  ↓
jq で集約・俯瞰データ作成
```

親 Claude Code は最後の俯瞰データだけ見て、コードベースの構造を把握する。全関数本体を親のコンテキストに載せる必要がない。

### 例 A-2: エラーハンドリングパターンの棚卸し

`try`/`catch`、`Result`、`throw`、`unwrap` などを含む箇所を抽出し、それぞれを分類:

- try_catch
- result_type
- throw_and_propagate
- unwrap_assert
- ignore

親 Claude Code がリファクタ計画を立てる材料になる。

### 例 A-3: 変更影響範囲の候補抽出

「`User` 型のシグネチャを変更したい」と親 Claude が判断した時、影響範囲の候補ファイル群を軽量モデルで事前評価する。「define」「consume」「test」「mock」「unrelated」のような分類を一括でつけさせ、親 Claude は consume の高確信度のファイルだけ重点的にレビューする。

## パターン B: ドキュメント・データの大量変換

### 例 B-1: i18n キーの一括翻訳

英語のローカライゼーションキー一覧を対象言語に翻訳する。キーごとに独立で、並列実行できる。完全に機械的な変換で検証可能(翻訳品質は別問題だが、動作としては安全)。

### 例 B-2: コミットログからのリリースノート草稿

数百件のコミットを対象に、それぞれを:

- カテゴリ(feat / fix / refactor / docs / chore / breaking)
- ユーザ向け度(boolean)
- 1 行サマリー

として構造化抽出する。親 Claude はこの構造化データを元にリリースノートを書く。

## パターン C: 評価・レビューのファンアウト

### 例 C-1: PR 内の全変更ファイルを独立レビュー

PR で変更された各ファイルについて、独立に以下の観点で評価:

- セキュリティ
- パフォーマンス
- テスタビリティ
- 可読性

各観点ごとに `{level: info|warn|critical, note: string}` で返させ、親 Claude Code は critical を重点的にレビューする。

### 例 C-2: テスト不足の検出

各関数と対応するテスト(があれば)のペアを対象に、テストの十分性を分類:

- well_tested
- needs_edge_cases
- no_tests
- trivial

親 Claude Code はテスト追加の優先順位を決める。

## パターン D: データ前処理・クリーニング

### 例 D-1: ユーザー問い合わせの感情・カテゴリ分類

サポートチケット群を対象に、感情(positive/neutral/negative/angry)、カテゴリ(billing/bug/feature/howto/other)、緊急度(low/medium/high)を抽出。

### 例 D-2: スクレイピング結果からの構造化

HTML や PDF 由来のテキスト群から、指定したフィールド(title、author、published_at、price、tags など)を抽出する。

## パターン E: 検証・照合

### 例 E-1: リファクタ前後の意味等価性チェック

変更前後のコードペアを対象に、「意味的に等価か」「懸念があれば何か」を判定させる。親 Claude Code は等価でないと判定されたものを精査する。

### 例 E-2: 翻訳品質スコアリング

原文・訳文ペアを対象に、`{accuracy: 0-1, fluency: 0-1, issues: [string]}` のような構造でスコアを返させる。

## 共通パターン

気づくと、**全て `map` 操作**である。

```
入力(多数の独立した item)
  ↓ map: 軽量モデルで各 item を処理
中間結果(多数の構造化データ)
  ↓ reduce: 親 Claude Code が集約・判断(または人間が)
最終的な意思決定
```

`hunch` は map 部分を担当する。reduce は親 Claude Code が jq や awk などのユーティリティを組み合わせながら、必要なら自身の推論を挟みつつ行う。

## 使用の判断チェックリスト

親 Claude Code がこのツールを使うべきか迷った時のチェックリスト:

- [ ] タスクが 10 件以上ある
- [ ] 各タスクが独立(他の結果に依存しない)
- [ ] 1 件あたり 5 秒未満の軽い判断で済む
- [ ] 結果が構造化可能(ラベル / 短文 / JSON)
- [ ] 失敗しても再実行すれば済む(破壊的操作ではない)
- [ ] 親 Claude Code の文脈に全結果を載せる必要がない

**6 項目すべて YES なら剥がすべき**。いずれかが NO なら親 Claude Code が直接やるか、別の手段を検討する。

## アンチパターン(やってはいけないこと)

### AP-1: コード変更

`hunch` で変更パッチを生成させない。整合性・責任・可観測性の問題がある。詳細は `docs/DESIGN.md`。

### AP-2: マルチターン必要なタスク

`hunch` は単ターンの map 専用。対話が必要なら親 Claude Code で。

### AP-3: 少数件(10 件未満)への適用

パイプ構築やプロンプト調整のオーバーヘッドが、節約されるコストを上回る。親 Claude Code が直接処理するほうが速い。

### AP-4: 結果を全件親コンテキストに戻す

「300 件分類して、全部 Claude Code に表示して」は意味がない。中間集約(jq で集計、サンプル抽出など)を挟むか、サマリーモード(F-7)を使う。

## 実装後に更新すること

具体的なコマンド例(フラグ名、引数形式を含む)は実装後にこのドキュメントに追記する。あるいは `docs/COOKBOOK.md` として別ファイルに分離する。実装前のこの時点では、**道具の使い方の概念**を記述するに留める。
