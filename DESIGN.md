# DESIGN.md

`hunch` の全体設計と、設計判断の背景。個別の判断に迷った時は、このドキュメントと `docs/context/design-discussion.md` を参照すること。

## 目的

Claude Code セッションで発生する「**大量・同質・軽量判断・独立**」という性質を持つタスクを、親 Claude Code の文脈とコストから剥がし、安価で高速な LLM(Gemini 3.1 Flash-Lite など)に並列でオフロードする。

## 名前の由来

`hunch` は「直感、軽い推論」を意味する。軽量モデルが出す「大量の小さな直感」をパイプで流し込んで並列に得る道具、というコンセプトの象徴。3 つの軸を 1 語で表している:

- **推論・思考** — hunch は軽い判断
- **軽量性** — hunch は深い推論ではない
- **並列展開** — パイプから多数の hunch を得る

「LLM」「AI」といった実装語を含まないので、将来 API や手段が変わっても名前が古びない。

## 解くべき問題

Claude Code にこの種のタスクを直接やらせると:

- **コンテキスト汚染** — 数百件の分類結果を全部メインコンテキストに載せると、以降が苦しい
- **コスト非効率** — Opus や Sonnet で単純分類は過剰
- **並列性の欠如** — 本体セッションは逐次的なので、大量ファンアウトと相性が悪い

これを軽量モデルに剥がすことで、Claude Code 本体は意思決定とコード変更に集中できる。

## スコープ

### やる

- 同質なタスクの **map 操作** を並列実行する
- **Unix パイプフィルタ** として振る舞う(stdin / stdout 基本)
- 読み取り・分類・抽出・要約・翻訳・変換を扱う
- 結果は構造化された形で返し、親 Claude Code がそれを俯瞰できるようにする

### やらない

- **コード変更は扱わない**。これは親 Claude Code の `edit` ツールで行う
- マルチターンの対話
- エージェント的なループ(外部 API 呼び出し → 再推論)
- 整合性が必要な連続操作(ファイル間の一貫性維持)

**なぜコード変更を扱わないか** — 整合性・責任・可観測性の問題があるため。詳細は `docs/context/design-discussion.md` の「責任分界」セクションを参照。

## アーキテクチャ

```
親 Claude Code(オーケストレーター)
   │
   │ bash ツールで以下のようなパイプラインを組む
   │
   ▼
<データ源> → 前処理(jq/rg/awk等) → hunch → 後処理(jq/awk等) → 親 Claude Code が集約
                                      │
                                      │ HTTPリクエスト(並列)
                                      ▼
                         OpenAI 互換エンドポイント
                         (Gemini Flash-Lite がデフォルト想定)
```

各層の責任:

| 層 | 責任 |
|---|---|
| 親 Claude Code | パイプラインの設計、前処理と後処理の指示、結果の統合・判断、コード変更 |
| `hunch`(Rust バイナリ) | stdin の JSONL を並列で LLM に投げ、結果を stdout に返す |
| Unix ツール(jq, rg, awk) | 前処理・後処理 |
| LLM API | 実際の推論(軽量モデル) |

## 設計判断の記録

### なぜ Gemini 3.1 Flash-Lite をデフォルトにするか

- 入力 $0.25/M、出力 $1.50/M で Haiku 4.5 の約 1/4 のコスト
- 2.5 Flash の 2.5 倍速、軽量タスクには十分な品質
- OpenAI 互換エンドポイントが公式提供されている
- 日本語タスクでも Google 多言語データの強み

将来のモデル切り替えは環境変数や CLI フラグで吸収できる設計にする。ただし実装時点ではデフォルトを決めて動くようにする。

### なぜ Skill 方式か(Anthropic 互換レイヤーではなく)

互換レイヤー(Ollama 方式)は Claude Code のサブエージェント機構を経由するが、軽量タスクには以下の問題が致命的:

- Claude Code のシステムプロンプト 18K+ トークンが毎回乗る
- Batch API の 50% 割引が使えない
- 並列度を自由に制御できない
- 実装コストが桁違い

Skill 方式は親 Claude が bash でパイプラインを組んで叩くだけ。オーバーヘッドが最小。

### なぜ Rust か(Bun/Python ではなく)

Unix パイプフィルタ化を決定した時点で、以下が要件になった:

- **ストリーミング処理** — stdin を逐次読み、stdout に逐次書く。`head` などで早期終了した時の SIGPIPE 対応
- **単一バイナリ配布** — Skill に置いて使うので、実行環境の依存が少ないほうがいい
- **起動速度** — パイプの中で何度も叩かれる可能性
- **並行処理の厳密性** — 多数の HTTP リクエストの並列制御

これらは Rust が最も素直に実現できる領域。要件とツールの強みが一致している。

採用するクレート想定:

- `tokio` — 非同期ランタイム
- `reqwest` — HTTP クライアント
- `serde` + `serde_json` — JSON パース
- `clap` — CLI 引数
- `futures` — 並行ストリーム処理
- `anyhow` または `thiserror` — エラー型
- `tracing` — ログ(構造化)

これら以外は原則として追加しない。CLI ツールの流儀に沿った薄い実装を保つ。

### なぜ最初から Rust か(Bun で始めて移行せず)

通常は動的な要件に対応しやすい言語で始めてから移行するのが定石だが、今回は例外:

- **要件の核(パイプフィルタ、JSONL、並列制御)は既に固まっている**
- **実装作業自体は Claude Code に任せる** — 人間の Rust 習熟度は実装速度の律速にならない
- **移行コストを避ける** — Bun → Rust の書き直しは、いくら仕様が明確でも非自明な手戻り
- **Rust の弱点(反復サイクルの遅さ)は Claude Code 運用で緩和される** — コード生成 → テスト → 修正のループがプログラマ一人のキーボード速度に依存しない

ただし以下の項目は **実装時に決める**(`docs/INTERFACE.md` の方針に従う):

- 具体的な CLI フラグ名
- 入出力 JSONL の詳細スキーマ
- タスクプリセットの具体的なプロンプト内容
- 並列度のデフォルト値

これらを事前に固定すると、実装中の試行錯誤が制限される。方針だけ決めて、具体は Claude Code が実装しながら最適化する。

### なぜ軽量 LLM に剥がすのは「読み取り系」に限定するか

コード変更を並列化する誘惑はあるが、構造的に不整合が出る:

1. 分類の失敗は統計的に許容できる(評価セットで一致率を測れる)
2. コード変更の失敗は離散的(1 ファイルでもビルド壊したら全滅)
3. ローカルに閉じないコード変更(シグネチャ変更、命名統一)は並列実行で整合性が取れない
4. 責任と可観測性を親に集約する必要がある

詳細は `docs/context/design-discussion.md` の「責任分界」セクション参照。

## パイプフィルタとしての作法

`hunch` は Unix の作法に従う:

- **stdin で入力を受ける**(`--input` フラグでファイル指定も可だが、stdin がデフォルト)
- **stdout に結果を書く**(`--output` フラグでファイル指定も可)
- **stderr に進捗・エラーを書く** — stdout を汚さない
- **SIGPIPE を正しく処理する** — 下流が終了したら速やかに停止
- **行指向で逐次処理する** — 全件読み込んでから処理する実装は避ける
- **終了コード** — 全件成功は 0、致命的エラーは非ゼロ。個別失敗は別ストリームに流して全体は継続

## スコープの将来拡張

以下は将来スコープ内だが、運用で必要性が見えてから実装する。詳細は `docs/REQUIREMENTS.md`:

- レジューム(途中から再開)
- コスト見積もり(`--dry-run`)
- サンプリング(`--sample N`)
- キャッシュ
- Batch API 対応(50% 割引)
- 評価モード(複数モデル横並び)
- プロバイダー抽象化

これらを MVP に含めない。MVP は「パイプフィルタとして動く最小のもの」に留める。

## 将来の退避先(捨てる場合)

このツールが不要になる状況もありえる:

- Claude Code 自体が軽量モデルサブエージェントを正式サポート
- Anthropic が非サブスク経路で Batch API を Haiku に開放し、Haiku のコスト優位性が Flash-Lite を上回る
- MCP サーバーとして同等機能を提供する上位互換ツールが現れる

これらが起きたら、このツールは deprecate する。`docs/context/design-discussion.md` に議論の経緯を残してあるので、将来の判断材料になる。
